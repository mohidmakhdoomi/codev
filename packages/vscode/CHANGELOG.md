# Change Log

## [3.2.2] - 2026-07-02

### What's new

- **Agents view: three group-by axes (stage, area, architect) with a cycling toggle.** The Builders tree is now called Agents. Its title-bar button cycles between three grouping axes instead of two: stage (the default action axis — `SPECIFY → PLAN → IMPLEMENT → REVIEW → PR → VERIFIED`), area (the domain axis matching the Backlog view), and the new architect axis (the ownership axis — groups by the architect that spawned each builder). The button's icon shows the axis you'll switch *to* on the next click (VS Code toolbar buttons have no pressed state, so the icon-as-next-target keeps the cycle obvious without one). In architect mode only architects with in-flight builders appear as group headers — childless architects don't clutter the work view; the full architect roster lives in Workspace > Architects. The architect axis ships with a custom theme-adapting octopus icon (one body, many arms — one orchestrator spawning many builders), monochrome light / dark to match the rest of the Codev sidebar vocabulary. View id renamed `codev.builders` → `codev.agents` to match; setting keys retain their `builders*` prefix to avoid forcing a user-facing migration.

- **Add Architect is now a conversation with main, not a direct CLI call.** The `Codev: Add Architect` action (`Cmd+K A` / `Ctrl+K A` or the command palette) no longer runs `afx workspace add-architect` directly. It now asks the `main` architect — the workspace orchestrator that owns backlog triage, release decisions, and architect-roster management — to create the new architect. VS Code prompts for the name (validated against the same rule Tower enforces server-side), then dispatches the request to main via `afx send architect:main`. Main decides whether the specialisation makes sense, runs the command from its own terminal, and briefs the new architect via the first message. The handler refuses with an informational modal pointing at `afx workspace start` (or the CLI fallback `afx workspace add-architect --name <name>`) when no main session is active — the action's contract is "ask main to add", so without main there's nothing to ask. Architect creation belongs in main's lane for the same reason cross-cutting work and release decisions do; letting any developer create an unbriefed architect from the sidebar `+` leads to roster drift.

- **Architect group-header click opens the architect terminal.** In the new Agents view's architect-axis grouping mode, clicking an architect group header (e.g. `MAIN`, `VSCODE`) opens that architect's terminal — parity with the existing builder-row click-to-open affordance. The expand-collapse chevron remains a separate target so the two gestures don't conflict. Only the architect axis gets the affordance; stage and area headers stay as pure grouping containers.

- **Add review comments from the Codev Markdown Preview without leaving the block.** The `+` affordance next to a rendered block in the Codev Markdown Preview no longer flashes a top-of-window Quick Pick to collect the comment body. It now opens an inline composer anchored *below the block itself*, keeping the anchor visible while you type. Multi-line input, `Enter` to submit, `Esc` to cancel, focus restoration on dismiss. The composer inherits the preview's prose typography so the input reads like the document it's commenting on. New comments on a block that already has a thread now write at the bottom of that thread (newest-last in reading order), matching the composer's visual position — previously the write went to the top of the thread while the composer appeared below.

- **Edit and delete review comments — the Markdown Preview reviewer is now fully functional.** Review comments in the Codev Markdown Preview now carry per-card edit and delete affordances, and comments can also be edited from the editor's Comments-API thread via a pencil on each comment. Click edit to reopen a comment's body in the same inline composer (pre-filled with the current text); click delete (preview side) to remove the marker in place. Both surfaces preserve the comment's original author — the author is read from the on-disk marker and never overwritten by whoever performs the edit. Every edit and delete runs an optimistic-concurrency check first: if the file changed since the preview last rendered (the comment moved, or a sibling comment on the same block was edited), the write is refused and the preview refreshes instead of clobbering the wrong comment, so stacked comments on one block are always addressed individually. Together with the inline composer shipped earlier this cycle, you can now author, edit, and remove review feedback entirely from the rendered preview without dropping to the raw markdown.

- **Open a specific issue by typing its ID.** New `Codev: Open Issue by ID...` command bound to `Cmd+K I` / `Ctrl+K I`. Prompts for a numeric issue identifier (accepts `1234` or `#1234`, with whitespace tolerance), fetches the issue via Codev's forge-agnostic path, and opens its canonical page in your browser. Works on any issue — open, closed, archived, or one you've never had loaded in the backlog. Cross-forge: the GitHub, GitLab, Gitea, and Linear adapters now emit an issue `url` field, so the same command opens the right canonical page regardless of which forge your workspace is bound to. Falls back to Codev's in-editor issue preview when a forge doesn't supply a URL. The existing `Codev: Search Backlog` Quick Pick is untouched — it remains the right choice for filtering across the loaded backlog set; `Open Issue by ID` is the direct-input companion for when you know the exact number.

- **Activity hooks — publish window-focus and builder-active events to URLs you declare.** A new `activityHooks` config lets the extension publish two abstract events to URL templates declared in personal config: `window-focus` (fires when a Codev workspace's VS Code window gains focus) and `builder-active` (fires when you switch focus to a specific builder via terminal, sidebar selection, or its diff editor). Each event interpolates `{workspace}` and `{builder}` into the URL template (URL-encoded) and asks the OS to open the result, so anything that can register a URL scheme — a hardware controller, a desktop companion app, a hosted control page, a webhook receiver — can react to which workspace and which builder you're focused on. Hooks resolve from personal config layers only (`~/.codev/config.json` global, or `.codev/config.local.json` per-project) — never from the committed `.codev/config.json`, so a hostile repo can't ship a URL that runs on workspace open. The extension stays fully disabled in untrusted workspaces (VS Code Restricted Mode) and routes URL opens through `vscode.env.openExternal` so deep links forward to the local client under remote dev (SSH / WSL / Codespaces) instead of the extension host. `builder-active` is de-duplicated across the three subscription sources so rapid navigation within one builder doesn't relaunch the same URL. Inert by default — no hooks declared means zero behaviour change.

- **External-controller command relay and four new commands.** The VS Code extension now exposes a Tower-driven command channel: external controllers (a Stream Deck device, a hosted page, any HTTP poster) can POST a verb to Tower's `/api/command` and the focused VS Code window dispatches it to an allowlisted command. The relay is workspace-scoped (the provider drops events whose `workspace` field doesn't match the active workspace, mirroring the `builder-spawned` cross-workspace filter), focus-gated (only the focused window runs a relayed verb, so multi-window setups execute exactly once), and allowlisted (only declared verbs reach the provider). Four new commands ship in this PR's verb set and are independently usable from the command palette and keyboard: `codev.forwardCurrentFileToBuilder` (cursor's current file to the builder PTY), `codev.forwardCurrentHunkToBuilder` (cursor's current hunk to the builder PTY), `codev.diffFirstFile` and `codev.diffFirstHunk` (jump to the first file or first hunk in the View Diff session, completing the existing keyboard walk from PR #1067 and #1075).

## [3.2.1] - 2026-06-21

### What's new

- **Add Architect: a UI button + keyboard shortcut for registering more architects in this workspace.** The Workspace tab's Architects section was already multi-architect-aware (it lists every registered architect from Tower's state), but adding a new one required dropping to the CLI (`afx workspace add-architect --name <name>`). A new `Codev: Add Architect` command surfaces as a `+` action on the Architects row and as a `Cmd+K A` / `Ctrl+K A` keyboard shortcut; both flow through a Quick Pick prompt for the architect name. Architect names render in UPPERCASE in the sidebar (MAIN, OB-REFINE, etc.) so they read as a name family at a glance, distinct from the file/folder labels around them. The architect-name validator that was duplicated between the CLI and the extension is now a single source in `@cluesmith/codev-core`, so name validation is identical no matter which surface the user adds an architect from.

- **Builders sidebar reveals and selects the active builder-file diff.** Opening a builder's per-file diff (sidebar click, multi-file View Diff, keyboard nav with `Ctrl+Alt+]` / `Ctrl+Alt+[` from PR #1067) now auto-reveals the corresponding file row in the Builders sidebar and selects it, mirroring how the Explorer tracks the active file. One listener covers every entry point; a registry-change re-reveal handles the case where the diff editor was opened before the changed-file list finished loading. File-row navigation now follows the visible tree order (depth-first in tree mode) and wraps at the ends to match VSCode's built-in hunk navigation. A new opt-out setting `codev.buildersAutoReveal` (defaulting on) disables the reveal for users who prefer a static sidebar. The reveal is gated to diff tabs only so a plain editor open of a tracked worktree file does not hijack the sidebar selection.
- **Typography controls for the Codev Markdown Preview.** Prose in the preview no longer inherits the workbench code font and tight code-tuned line-height; it now renders with prose-readable defaults pinned to `github-markdown-css` v5.8.1 (font-size, font-family, line-height, paragraph spacing, optional prose-width cap, per-heading scale, code font / size). Block elements that were falling through to user-agent defaults (inline `code` chips, fenced `pre`, blockquotes, tables, `hr`, images, list indentation) are styled too. Code tokens bind to VSCode's editor font while prose keeps the readable sans stack; inline code pairs with VSCode's theme-tuned `textPreformat-*` tokens so dark themes render proper contrast. Two new settings, `codev.markdownPreview.fontSize` and `codev.markdownPreview.lineHeight`, let you tune the preview independently of the rest of the IDE chrome; both reflow live on change. The same typography also covers the package's standalone `MarkdownView` surface, not just the composed `ArtifactCanvas`, so any host mounting the canvas gets consistent rendering. A page-level horizontal-scroll bug from unbreakable long tokens in prose is fixed by `overflow-wrap: break-word` on the prose container (with `pre` / tables keeping their own `overflow: auto`).
- **Cross-file navigation in the Codev View Diff session.** Two new commands, `codev.diffNextFile` and `codev.diffPreviousFile`, walk a builder's changed-file list and open the next or previous file's per-file diff. Bound to `Ctrl+Alt+]` / `Ctrl+Alt+[` by default and palette-discoverable, scoped via the new `codev.activeEditorIsBuilderFile` context key so the keybindings only fire while you're sitting on a Codev builder-file diff (not in unrelated VS Code diff editors). The keyboard equivalent of clicking the next file row in the Builders sidebar, taken from GitHub PR review's `j` / `k` muscle memory. The nav anchor is seeded on every diff open, including sidebar clicks for deleted or binary files, so a file opened directly from the sidebar can be the starting point of a walk. Reuses the same `BuilderDiffCache` that already backs the sidebar's changed-file list, so the navigation order matches what the sidebar shows.

### Bug fixes

- **Backlog issue preview no longer steals focus to group 1 and cannot chain into new editor groups.** Clicking a backlog row in the Codev sidebar used to focus the first editor group before opening the preview, pulling the user away from wherever they were (e.g. a builder terminal in group 2). The placement also depended on the `Beside` resolver's active-group-relative semantics, so if any future caller skipped the focus step the preview would land in group 3, then 4, etc. The preview now opens via `vscode.openWith` with the built-in markdown preview's custom-editor viewType and an explicit `viewColumn` (group 2 when one exists, else group 1, matching the builder-terminal count-then-pick model). Focus stays on the row you clicked. No change to which renderer draws the preview: this still uses VSCode's built-in markdown preview.

## [3.2.0] - 2026-06-17

### What's new

- **Markdown preview shows review comments in-context and a marker map down the right edge.** Two related additions to the canvas-powered preview that opens specs / plans / reviews. Inline comment cards now anchor *below* each annotated block instead of overlaying it, so they no longer hide the content they comment on. The card stack carries a thin left rule tying back to the gutter where the `+` action lives, so the visual association between a card and its block stays clear. A right-edge minimap shows a colored dot for every comment in the document, positioned proportionally to where the comment lives in the preview viewport; hovering shows author + body, clicking smooth-scrolls to it. List, blockquote, and table comments now render exactly one card stack per source line (previously rendered duplicates and could produce invalid nested DOM). The preview becomes the complete review surface: read, see existing feedback in context, see where feedback is concentrated at a glance, click to navigate.
- **Add review comments from a rendered Markdown Preview.** A new `codev.openMarkdownPreview` command opens specs, plans, and reviews in a host-owned preview powered by the `@cluesmith/codev-artifact-canvas` package (shipped earlier this cycle as #945). Hover any rendered block, click `+`, type your feedback, and a `<!-- REVIEW: author "body" -->` marker is written above the block — the same on-disk convention the editor's Comments-API thread already uses, so the two surfaces are bit-compatible. Markers no longer render as visible HTML comments (the canvas renderer strips them pre-parse) and multi-line blocks no longer split around marker lines. Safe inline HTML (`<img>`, `<details>`, `<kbd>`, tables) renders via DOMPurify; scripts and `javascript:`/`data:` URLs are stripped. Registered with `priority: "option"` so it never replaces the default `.md` editor or the built-in markdown preview — opt-in via `Reopen With…` or the command. First VSCode consumer of artifact-canvas; the dashboard host follows in a later cycle.
- **Pull Requests sidebar carries an inline `Reference PR in Architect` action.** Mirrors the existing backlog row action: clicking the `$(mention)` icon on any PR row injects `#<pr-number> "<title>" ` into the architect's prompt buffer (no Enter pressed), opening the architect terminal first if it isn't already. Quotes embedded in PR titles are escaped. The same action is reachable from the row's right-click context menu. Closes the asymmetry that left PR rows without a fast hand-off to the architect.
- **Codelens in the unified diff editor injects file / hunk references into the builder PTY.** When you open a builder's `View Diff`, each file header now carries a `> Send to builder PTY` codelens and each hunk header carries `> Send to builder PTY (lines N-M)`. One click injects `path/to/file.ts ` or `path/to/file.ts:L42-L58 ` into the builder's prompt buffer (no Enter pressed). You type your feedback after the injected reference and submit when ready. Closes the "type the file path by hand during review" friction that was the dominant slow step in the architect-to-builder feedback loop. The builder is taken from the diff's context (no picker), Cmd/Ctrl+K B is the keyboard shortcut for the same action, and a right-click menu entry is available on builder files in the file tree as well.
- **PR sidebar sorts by ownership, with a `(draft)` badge.** The Pull Requests view groups PRs into one flat list ordered mine first, then review-requested, then others, newest-first within each bucket. Drafts carry a `(draft)` suffix and a draft icon so you can tell them apart at a glance. When `gh` isn't available the list falls back to plain newest-first ordering.

### Bug fixes

- **Terminals no longer render corrupted on open.** A terminal could come up with stacked, overlapping status lines and the cursor landing near the top of the pane instead of at the prompt, fixable only by resizing the window, clearing, or reopening the tab. On open, the editor reports the pane's real size in two quick steps; the terminal used to paint the restored on-screen history immediately at the first (not-yet-final) size, so a full-screen program's frame was laid out at the wrong width and a stale copy was stranded in the scrollback. The terminal now holds the restored history until the size has settled and paints it once, at the final width — the same approach the web dashboard already uses. A new opt-in setting, `codev.terminal.repaintOnRefocus` (off by default), additionally forces a redraw when the VSCode window regains focus, for the rare setup that still sees stale content after switching back to the window.
- **Builder terminals correctly indicate "session exited" instead of staying stuck as active.** When a builder's PTY exited during the brief disconnect window of a Tower restart, or when you reopened a builder tab whose terminal had already died, Tower's session tracking could miss the exit signal and leave the session permanently marked active. The VSCode Builders sidebar then showed a builder that wouldn't go idle and the terminal pane never displayed an exit indicator. Tower's shellper protocol now replays the exit signal to clients that connect after the PTY has already exited, so Tower's tracking stays correct and the UI reflects reality. Mostly affects Tower restart and rapid reopen scenarios; in normal long-running interactive sessions the existing exit broadcast already worked.
- **Builder and architect terminals no longer freeze until a Tower restart.** Full-screen TUIs (like Claude Code's UI) redraw in place and emit almost no newlines, so Tower's reconnection buffers — sized by *line count* — grew unbounded for these sessions. The resulting multi-megabyte replay overflowed the VSCode client's 1 MB receive budget, the client responded by disconnecting and reconnecting to re-fetch the snapshot, which was still oversized — repeating ~14k cycles per hour, pegging one CPU core and starving every other terminal of output. Tower now scans only new output (killing an O(n²) re-scan) and brackets reconnection replay with `pause`/`resume` control frames so it's paced and excluded from backpressure. The VSCode client drops live output under overload instead of reconnecting in a loop, reconnects with a `?resume=<seq>` delta where the stream has newlines, and forces a guaranteed redraw shortly after attaching so the TUI repaints cleanly. Re-attach now also drops the previous client's `data`/`exit`/`close` listeners so a stale frame can't reach the buffer. Long-running TUI sessions stay responsive across many days of uptime; a new partial-size monitor logs ring-buffer pressure so growth is observable. Also fixes the blank-pane-on-open regression where a same-size connect-time resize produced no redraw signal.
- **Spawning a builder terminal no longer force-creates a second editor group.** The builder/shell terminal spawn used to unconditionally target `ViewColumn.Two`, which is fixed by ordinal — so when only one editor group was open, VS Code conjured a second group on demand and reshaped the user's layout every time a builder spawned. The spawn now picks `Two` only when a second tab group already exists; single-column users get the terminal in their first/default group instead. Architect terminals (`ViewColumn.One`) and dev/panel terminals are untouched.
- **Expanding a builder in the sidebar no longer collapses area-group headers.** The Builders tree's accordion (expand one builder, collapse the others) was firing `workbench.actions.treeView.codev.builders.collapseAll`, the bluntest available API, which collapses every expandable node in the tree — including the `VSCODE` / `TOWER` / etc. area-group headers. Combined with the area-group expansion persistence, every accordion click also wrote `false` for every area-group to `workspaceState`, so the collapses survived reloads and you had to re-expand each group manually on every restart until you next interacted with it. The accordion now only touches sibling builder rows, area-group expansion in the Builders view is in-memory-only (Backlog's persistence is unchanged — different lifecycle), and any previously-persisted Builders group state is cleared once on activation so the dead state doesn't linger forever. Toggling the accordion title-bar button while one builder is open also now correctly resets so the next expand on any builder, including the previously-open one, collapses the rest.
- **CLI preflight no longer triggers a false "Get started with Codev" walkthrough on slow environments.** The startup CLI version probe used to cap at 400ms — too tight against the realistic ~500-3500ms cold-spawn budget on remote SSH, WSL, `nvm` / `fnm` / `volta` shims, AV-scanning Windows, and network filesystems. A timed-out probe was wrongly deciding the CLI was missing, re-opening the walkthrough on every startup and no-op'ing every guarded command even though `codev --version` succeeded from a terminal in the same window. The cap is now 5000ms by default and overrideable via a new `codev.cliVersionTimeoutMs` setting (range 100-60000ms) for users on extra-slow infrastructure. Timeouts now log a `[Preflight]` line to the Codev Output channel so the failure mode is diagnosable instead of silent.

## [3.1.8] - 2026-06-08

### What's new

- **Tower version preflight: warn when the running Tower is behind the installed CLI.** The v3.1.7 CLI preflight verified that the installed `codev` CLI was current; this release closes the matching gap on the running side. After an `npm install -g @cluesmith/codev` upgrade without a Tower restart, the installed binary is the new version but the running Tower process is still serving stale handlers from the old code. The extension now probes Tower's in-memory version on connect and shows a `Restart Tower` toast when the two diverge. One click runs `afx tower stop && afx tower start`, polls until Tower comes back up, and re-probes to confirm. The healthy path stays silent (no toast when running matches installed). The CLI-row tooltip in the Status view surfaces both versions alongside each other.
- **Codev Dev surface: bottom-panel tab + status-bar chip.** A new `Codev: Dev` tab in the bottom panel and an always-visible status-bar chip (`$(server-process) Dev: <target>`) make the running dev server visible at a glance. The tab shows target name, live-ticking uptime, and best-effort port; title-bar actions for Stop, Restart, Switch Target, and Show / Hide the Codev sidebar. The chip appears only while a dev is running and clicks through to the tab. The native `Codev: <name> (dev)` terminal stays as the output surface, with the tab and chip layered on top as status surfaces (no PTY re-plumbing).
- **Codev tab in the bottom panel.** A new `Codev` tab joins the bottom panel (alongside Problems / Output / Terminal) as a scaffold for upcoming view migrations (Recently Closed, Team, Status). Opens once on first activation for discoverability, then stays out of the way. The activitybar Codev sidebar is unchanged; this is purely additive. Currently shows a single placeholder row that hides itself automatically once real panel-side views start landing.

### Bug fixes

- **"No active terminal" toast now self-heals and offers a recovery action.** The `Codev: No active terminal for X` toast used to fire once per session as a dead-end warning, then go silent on retry, even when Tower was actively reconnecting. Terminal resolution now runs a short bounded retry before raising any UI, so transient reconnect windows self-heal silently. When retries genuinely fail, the toast carries `Retry` (reattempt) and `Recover Builders` (opens a terminal at the main checkout running `afx workspace recover --dry-run` so the scope can be reviewed before applying — recover is workspace-wide and can revive builders the user didn't mean to touch). The recover terminal's cwd is resolved through the main checkout root even when VSCode is rooted at a builder worktree window, so `afx` runs where it must.
- **Builders no longer briefly flash into `UNCATEGORIZED` during cleanup.** When you cleaned up a builder, the Builders tree used to re-render it under the `UNCATEGORIZED` group for a few seconds before it finally disappeared. The area was re-derived every refresh from the open-issues list, so once the builder's issue was unreachable (closed on PR merge, torn down mid-cleanup, or a failed fetch) the lookup missed and the row fell back to its `Uncategorized` default. The row's `area` now memoizes the last *resolved* value while the issue is reachable, so the builder stays in its real group until the cleanup completes and it disappears for real. No regression for builders whose issue genuinely has no `area/*` label (those continue to live under `UNCATEGORIZED` throughout).
- **Terminal reconnect notice overwrites in place and wipes on reconnect.** The `[Codev: Connection lost. retrying in Xs (attempt Y/6)]` line used to print one fresh `\r\n`-terminated line per retry attempt, stacking notices in scrollback and leaving them orphaned above the resumed output once Tower came back. The line now overwrites itself in place (single line ticking `1/6 → 6/6`) and is cleared the moment the WebSocket reconnects, so a healthy reconnect leaves no trace in the terminal. The red give-up notice (after 6 failed attempts) is unchanged: it stays visible as the terminal's failure state and still carries the clickable `Reconnect` token.
- **Guarded commands always give feedback now; the silent-after-first-click behavior is gone.** Clicking a CLI-dependent command (Spawn Builder, Approve Gate, Send Message, and 12 others) while the Codev CLI is missing or outdated used to produce a modal toast on the first click of the session, then go completely silent on every subsequent click. The first-click modal stays the same (the `Run Setup` action still works); subsequent clicks now show a brief auto-dismissing status-bar message naming the state and pointing at `Codev: Recheck CLI` as the recovery path. Once a recheck confirms `ok`, the next breakage restarts at the modal again.

## [3.1.7] - 2026-06-03

### What's new

- **Codev CLI preflight on startup.** The extension now verifies that the `codev` CLI is installed and at a version at least as new as itself. Missing → opens the new `Get started with Codev` walkthrough (detect / install / verify). Outdated → upgrade notification with one-click `Update via npm`. A persistent **Codev CLI** row in the sidebar's Status view shows the current state with a recheck button.
- **Builders tree groups by stage by default.** Lifecycle stage (`SPECIFY → PLAN → IMPLEMENT → REVIEW → PR → VERIFIED`) is the new default grouping axis for the Builders tree; `area/*` grouping is still supported via a toggle. In stage mode each row reads `[<area>] #<id> <title>`; in area mode the prefix flips to `[<phase>]`. Toggle the axis from either the title-bar (`$(tag)` ↔ `$(milestone)`) **or a control on the group header itself** — both flip the grouping and the row prefix together, and the choice persists per workspace via the new `codev.buildersGroupBy` setting. Backlog grouping is unchanged.
- **Roll-up status icons on `area/*` group headers.** Each area header in the Backlog and Builders trees now carries a status glyph so you can triage at a glance without expanding the group. Backlog: filled-grey dot = area has a builder, outline-grey = open to spawn. Builders: worst-of-three across the group — `bell` (any blocked) / `comment-discussion` (any idle) / `circle-filled` (all active), reusing the row vocabulary. Counts live in the tooltip.
- **Search Backlog editor-tab webview.** A new 🔍 icon in the Backlog view's title bar opens a full editor-tab panel for exploratory triage. Filter by Area / Assignee / Author, substring-search title + body, sort any column, see a match-count footer. Rows open the issue or reference it in the architect chat. Complements (rather than replaces) the always-on sidebar tree and the `Codev: Search Backlog…` Quick Pick.
- **`[new]` prefix on freshly-created backlog rows.** Backlog items created in the last 24 hours now lead with a `[new]` prefix right after the issue number (e.g. `#930 [new] vscode: mark recently-created...`). The prefix drops off automatically as items age past 24h — no per-user "mark as seen" state, no manual dismissal.
- **Click-to-reconnect affordance on terminal give-up.** When a Codev terminal hits its bounded-retry give-up state (the corresponding reconnect-overhaul fix lives under Bug fixes), the red give-up line now carries a clickable `Reconnect` token. One click starts a fresh retry chain from the same terminal — no more close-tab + re-open-from-sidebar dance to recover from a Tower restart or a forgotten session. The click targets the terminal whose line was clicked, not whichever Codev terminal happens to be active when there are multiple open. Repeatable across cycles: if the retry exhausts again, the affordance is still there.

### Bug fixes

- **Terminal reconnect no longer spams "Connection lost" indefinitely.** When the WebSocket backing a Codev terminal closes, the adapter now retries with an exponential backoff (1s → 2s → 4s → 8s → 16s → 30s), prints one line per attempt instead of a wall of identical notices, gives up cleanly after 6 attempts with a red failure line, and fast-paths a permanent "session no longer exists" failure (Tower restart) so you don't wait through 6 retries for an unrecoverable case.
- **Run / Stop Dev Server entries hide when `worktree.devCommand` isn't configured.** The Workspace view always honoured this; the builder-row context menu, dev keybindings (`Cmd/Ctrl+Alt+R`/`S`), and command palette didn't — they would appear on every blocked builder regardless, then fail with a missing-command error on click. All four surfaces (context menu / keybindings / palette workspace-dev / palette builder-row dev) now share one `codev.hasDevCommand` context key that refreshes live on Tower connect + config-update events. Also fixes a latent Workspace-view bug where `"devCommand": ""` would still surface the Start row.
- **Codev sidebar no longer flickers empty during transient connection blips.** The data-bearing views (Builders, Backlog, Pull Requests, Recently Closed) used to all blank simultaneously while the Workspace view stayed populated, then recover on their own. The shared overview cache was overwriting populated data with `null` on transient reads (not-connected state or a failed `/api/overview` fetch); every provider rendered the falsy cache read as an empty list, blanking all four views at once. The cache now holds last-known-good data — only successful fetches commit, and it re-syncs on reconnect. Trade-off by design: a long Tower outage shows *stale* fleet data rather than blanking. A separate "disconnected" visual treatment for the data-views is out of scope.
- **Blocked-builder inline action button no longer reads as "approve."** The `✓` glyph on the inline button (which appears for every blocked builder regardless of which gate they're at) is now a neutral `→` arrow. Behaviour is unchanged — clicking still opens the approve flow — but the icon no longer over-promises "approve" on rows whose underlying gate action is View Plan, Run Dev, etc. Per-gate triage continues to live on the row's leading icon.
